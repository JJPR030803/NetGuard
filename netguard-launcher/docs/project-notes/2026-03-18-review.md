## Resolved: State Machine Alignment

The `SystemState` enum has been redesigned to match the spec. The old 6-variant
flat enum (`Initializing, Ready, Running, Degraded, Stopping, Stopped`) was
replaced with the correct 8-variant enum carrying structured data:

```rust
pub(crate) enum SystemState {
    Initializing,
    CheckingEnvironment,
    Connecting,
    Ready,
    Operating { operation: ActiveOperation },
    Degraded { reason: DegradedReason, recovering: bool },
    ShuttingDown,
    Fatal { reason: String },
}
```

Also implemented: `ActiveOperation` (3 variants), `DegradedReason` (6 variants),
`can_transition_to()` with full transition table, `allowed_commands()` per state,
`Display` impl, `is_terminal()`, `is_degraded()`. 43 state machine tests.

Stale references in `orchestrator/error.rs` (doc examples and tests using
`SystemState::Running` / `SystemState::Stopped`) were fixed to use valid variants.

## Decisions & Observations

1. Error system quality is thesis-grade — no rework needed.
2. State machine is now fully aligned with spec — complete with tests.
3. The `orchestrator/error.rs` stale references to `Running`/`Stopped` have
   been fixed. Doc example changed from `no_run` to `ignore` because
   `SystemState` is `pub(crate)` and doc-tests compile as external crates.

5. Pre-existing clippy warnings remain: dead_code (pub(crate) types not yet
   consumed outside tests) and private_interfaces (`pub` error enum fields
   using `pub(crate)` `SystemState`). These will resolve as Phase 1 progresses
   and the orchestrator module consumes these types.
